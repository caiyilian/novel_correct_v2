# novel_correct_v2 — 开发方案

## 一、回顾：v1 的问题与 v2 的设计方向

### v1 的 5-Stage Pipeline

```
预处理 → Stage1 暴力扫描 → Stage2 规则定位 → Stage3 中括号清理 → Stage4 规则兜底 → Stage5 长对话校验
```

**v1 的核心问题**（与 novelSpeakLabel v1 一样）：

| 问题 | 表现 | 后果 |
|------|------|------|
| 滑动窗口无差别扫描 | 每 800 字符一个窗口，全部调 LLM | 大量无错误的窗口浪费算力 |
| LLM 是被动分类器 | Python 切好喂给它，LLM 只能看固定窗口 | 上下文不够时只能猜 |
| 无法主动翻书 | LLM 不能要求更多上下文 | 漏修、错修 |
| 多模型共识成本高 | 4 个模型 × 每个窗口 | 速度慢、token 消耗大 |
| 无断点续跑 | Stage 跑完才能进下一个 | 中断后重头再来 |

### v2 的设计方向（借鉴 opencode-novel-loop v2 的成功经验）

```
v1（被动分类器）：
  Python 切好小块 → 塞给 LLM → LLM 在小块里找 → 不够 → 猜

v2（主动代理）：
  规则检测定位错误 → LLM 拿到位置 → 自己决定读哪里 → 不够 → 再读 → 够了 → 修正
```

**核心转变**：给 LLM 工具，让它自主导航文本，而不是 Python 预先切好喂给它。

## 二、核心原则（优先级排序）

### 原则 1：质量 > 一切

本项目的唯一追求是纠错准确率。速度、token 消耗、运行时间通通不考虑。
- 一卷小说跑几天甚至几周都可以接受
- 只要能修得更准，多调几次 LLM 也没问题
- 宁可不修，不能修错

### 原则 2：统一所有对话符号为「」

**最终目标：小说中所有对话，无论原始使用什么符号包裹，全部统一为成对的全角直角引号「」。**

经过对真实数据集的抽样验证，原始文本中的对话包裹符号五花八门：

| 符号类型 | Unicode | 示例文本 | 出现情况 |
|----------|---------|----------|----------|
| 直角引号「」 | U+300C / U+300D | 「你好吗？」 | ✅ 大多数卷已使用 |
| 弯引号“” | U+201C / U+201D | “你好吗？” | ⚠️ 部分卷混用（如第3卷全卷使用） |
| 英文方括号 [] | U+005B / U+005D | [你好吗？] | ⚠️ 部分卷有大量残留（如第1卷 128个[、164个]） |
| 中文方括号【】 | U+3010 / U+3011 | 【你好吗？】 | ⚠️ 偶尔出现 |
| 全角方括号［］ | U+FF3B / U+FF3D | ［你好吗？］ | ⚠️ 偶尔出现 |
| 无符号 | — | 你好吗？ | ⚠️ 部分对话缺失包裹符号 |

每种非标准符号的纠错策略：

| 源符号 | 处理方式 |
|--------|----------|
| `「」` | ✅ 保留，无需修正 |
| `“”` (弯引号) | 规则检测 → LLM 判断上下文 → 替换为「」 |
| `""` (ASCII 直引号) | 规则检测 → LLM 判断上下文 → 替换为「」 |
| `[]` (英文方括号) | 规则检测 → LLM 判断是否为对话 → 替换为「」 |
| `【】` (中文方括号) | 规则检测 → LLM 判断是否为对话 → 替换为「」 |
| `［］` (全角方括号) | 规则检测 → LLM 判断是否为对话 → 替换为「」 |
| 缺失符号（对话无包裹） | LongDialogueDetector 检测 → LLM 判断并补充 |

> **注意**：`[]` `【】` `［］` 除了包裹对话外，也可能用作旁注、注释、舞台指示等非对话用途。所以不能无脑替换，必须由 LLM 根据上下文判断是否真的是对话包裹。这也是为什么 v2 采用"规则定位 + LLM 判断"的策略——规则负责找出来，LLM 负责判断要不要改。

### 原则 3：LLM 只修不找

**LLM 绝不承担"发现错误"的任务。** 错误定位全部由规则检测器完成。

为什么？
- 让 LLM 在一卷小说里找所有错误 → 必漏、必错（这是 v1 滑动窗口的教训）
- 规则检测器是确定性的，不会漏、不会幻觉
- LLM 只做它最擅长的事：理解上下文、判断符号是否该修正

### 原则 4：一次只修一个错误

**每轮 Agent 循环只处理一个 ErrorRecord。** 修完一个 → Verifier 确认 → checkpoint 持久化 → 再修下一个。

为什么？
- 一次处理多个错误 → LLM 容易漏改、改错、改串
- 逐个处理 → 每个修正都可以被精确验证
- 出错时只需回滚一个修正，不影响其他
- 中断后可以从 checkpoint 精确恢复，不丢不重

### 原则 5：验证过的修改才能落地

每个修正都必须经过 Verifier 确认才能写入。
- Verifier 验证不通过 → 自动回滚
- Verifier 不确定 → 记录到日志，人工后续审核
- 绝对不允许未经验证的修改写入原文

### 原则 6：宁跳过，不错改

如果 LLM 对一个错误不确定：
- 允许它调用工具读更多上下文
- 读了还是不确定 → 标记跳过，交给人工处理
- 绝不强行修改

## 三、终止条件设计

### 3.1 为什么纠错的终止条件比角色标注更难定义

角色标注的终止条件天然清晰：

```
全文有 N 个「」对话 → 标注了 N 条 → ✅ 完工
```

纠错则不同。即使 `「` 和 `」` 数量相等了，也不代表没有错误了：

| 场景 | 「数 | 」数 | 数量平衡？ | 还有错误吗？ |
|------|------|------|-----------|-------------|
| `[你好]「我很好」` | 1 | 1 | ✅ 平衡 | ✅ 还有 `[` 没修 |
| `「你好。他走上前来。你好。」` | 1 | 1 | ✅ 平衡 | ⚠️ 可能漏了符号，旁白被吃进对话了 |
| `「你好。\n「我很好」` | 2 | 1 | ❌ 不平衡 | ✅ 肯定有错 |
| `你好吗？`（无符号） | 0 | 0 | ✅ 平衡 | ⚠️ 可能是一句对话没加符号 |

所以终止条件不能只看数量平衡。

### 3.2 v1 的终止条件及其问题

**v1 的流程是一个直通管线（straight pipeline）：**

```
Stage1 扫描 → Stage2 规则 → Stage3 中括号 → Stage4 规则兜底 → Stage5 长对话 → ✅ 结束
```

每个 Stage 内部自己管自己的终止：
- Stage 2/3/4: 内部循环直到「找不到更多错误」或「死结停滞」
- Stage 1: 滑动窗口扫完一遍就结束
- Stage 5: 处理完 Top K 长对话就结束

**v1 没有跨 Stage 的循环回溯：**

如果 Stage 3 把 `[` 修成 `「` 后，导致 Stage 4 又发现新的连续符号错误 → Stage 4 修完就结束了，不会回到 Stage 2 再扫一轮。

具体问题：

| 问题 | 说明 |
|------|------|
| **直通管线，无反馈** | Stage 3 的修改可能在 Stage 4 产生新的错误，但不会再回 Stage 2 |
| **死结处理简单粗暴** | 遇到死结就把 `[` 替换成全角 `［` 绕过，实际上放弃了修复 |
| **无总体校验** | 没有最终的"全量检测 → 确认零错误"的验收环节 |
| **Stage 5 的 Top K 不严谨** | 只检查最长的 K 个对话，但第 K+1 个也可能有问题 |

### 3.3 v2 的分层终止条件

v2 不再采用直通管线，而是闭环流程。终止条件分三层：

#### 第一层：硬性指标（必须全部满足）

| 条件 | 检测方式 | 说明 |
|------|----------|------|
| `「` 总数 == `」` 总数 | 全文扫描计数 | 最基本的不成对检查 |
| 不存在连续相同符号（`「「` 或 `」」`） | `ConsecutiveDetector` | 交替模式被破坏 |
| 不存在非标准符号（`[]` `【】` `“”` 等） | `WrongSymbolDetector` | 所有对话包裹已被统一 |
| 不存在因缺失符号导致的超长对话 | `LongDialogueDetector` | 对话/旁白边界正确 |

这四条全部通过，才算"文本格式正确"。

#### 第二层：收敛性指标

每轮 Phase 1 检测后，`ErrorQueue` 的**错误总数必须单调递减**。

```
轮次 1: 检测到 150 个错误
轮次 2: 修复了一批，重新检测 → 23 个错误 ✅ 收敛
轮次 3: 又修复了一批，重新检测 → 5 个错误  ✅ 收敛
轮次 4: 又修复了一批，重新检测 → 5 个错误  ⚠️ 停滞
```

如果连续 2 轮重新检测后错误总数不再减少 → 判定达到瓶颈，终止 Agent 纠错。

#### 第三层：最终验收

```
Phase 1 硬性指标全过
    + Phase 2 收敛停滞（连续 2 轮零新错误）
    = ✅ 终止
```

剩下的 `skipped` 和 `failed` 状态的错误记录到报告，供用户人工审核。

### 3.4 整体流程

```
Phase 0: 加载 & 归一化
    ↓
Phase 1: 全量规则检测 → ErrorQueue
    ↓
Phase 2: Agent 逐条修正
    ├── 修完一个 → Verifier 确认 → checkpoint
    ├── Verifier 不通过 → 回滚，标记 failed
    └── LLM 不确定 → 标记 skipped
    ↓ (ErrorQueue 空了)
Phase 3: 重新全量检测
    ├── 发现新错误 → 合并到 ErrorQueue → 回到 Phase 2
    ├── 零错误 → 再扫一次确认
    │    ├── 还是零错误 → ✅ 终止，生成报告
    │    └── 发现了 → 回 Phase 2
    └── 连续 2 轮错误数无变化 → ⚠️ 停滞终止
```

这样设计的好处：
1. **闭环** — 修完一轮重新扫，不会漏掉修 A 导致 B 的情况
2. **多维度** — 硬性指标 + 收敛性 + 最终验收，三层保障
3. **可追溯** — skipped/failed 都在报告里，用户可以审查
4. **防死循环** — 收敛停滞机制确保不会无限循环

## 六、核心模块详细设计

### 6.1 Phase 0: 文本加载与归一化

**`src/io/loader.py` — `TextLoader`**

```python
class TextLoader:
    def load(path: Path) -> TextDoc:
        # 1. 自动检测编码（UTF-8 → UTF-16 → GBK）
        # 2. 统一换行符 \r\n → \n
        # 3. 去除 BOM
        # 4. 可选：去除多余空白（v1 的预处理）
        # 返回 TextDoc(original_text, lines, encoding)
```

### 6.2 Phase 1: 规则检测层

每种检测器输出一个 `ErrorRecord` 列表，合并到 `ErrorQueue`。

**`src/core/error_record.py`**：
```python
@dataclass
class ErrorRecord:
    error_id: str            # 唯一 ID（如 "e-0001"）
    error_type: str          # consecutive / wrong_symbol / unpaired / long_dialogue
    line_number: int         # 行号（1-based）
    offset: int              # 在全文中的偏移量
    context_before: str      # 错误的左侧上下文（用于 LLM 判断）
    context_after: str       # 错误的右侧上下文
    original_text: str       # 错误的内容片段
    expected_pattern: str    # 期望的符号模式
    status: str              # pending / fixed / skipped / failed
    fix_applied: str = ""    # 实际应用的修正
    verifier_verdict: str = "pending"  # pending / pass / fail
```

**检测器列表（按优先级排序）**：

| 检测器 | 检测内容 | 优先级 |
|--------|----------|--------|
| `ConsecutiveDetector` | `「「` 或 `」」` 连续出现 | P0 |
| `UnpairedDetector` | `「` 或 `」` 数量不成对 | P0 |
| `WrongSymbolDetector` | `[` `]` `【` `】` `［` `］` `“` `”` `"` 等非标准符号 | P1 |
| `LongDialogueDetector` | 超长 `「...」`（可能漏符号合并了旁白） | P2 |

### 6.3 Phase 2: Agent 纠错循环

这是核心设计，借鉴 opencode-novel-loop 的 Agent Loop 模式。

#### 工具集（Tools）

LLM 可以调用的工具，定义在 `src/agent/tools.py`：

```python
class CorrectionToolset:
    def __init__(self, text_view: TextView, error_queue: ErrorQueue):
        ...
    
    def read_lines(self, start: int, end: int) -> str:
        """读取指定行号范围的原文。允许跨章节阅读。"""
    
    def read_offset(self, offset: int, context: int = 200) -> str:
        """以指定偏移为中心，读取前后上下文。"""
    
    def search_text(self, keyword: str, limit: int = 20) -> list[SearchMatch]:
        """在全文搜索关键词，返回匹配行号和内容。"""
    
    def get_next_error(self) -> dict:
        """获取下一个待纠错的 ErrorRecord（不含 fix_applied）。"""
    
    def apply_fix(self, error_id: str, start_offset: int, end_offset: int, replacement: str) -> dict:
        """在原文中执行替换修正。返回 diff 信息。"""
    
    def get_progress(self) -> dict:
        """返回当前纠错进度。"""
    
    def skip_error(self, error_id: str, reason: str):
        """标记为跳过（LLM 确认不是错误）。"""
    
    def revert_fix(self, error_id: str) -> dict:
        """回滚上一次修正。"""
```

**注意**：`read_lines` 和 `search_text` 直接复用 opencode-novel-loop 中 `DialogueIndex.search()` 和 `read_lines()` 的逻辑。`apply_fix` 是纠错特有的新工具。

#### Agent Loop 流程

```
while error_queue.has_pending():
    # 1. 获取下一个错误
    error = tools.get_next_error()
    
    # 2. 构建 user prompt，包含错误信息和上下文
    prompt = build_correction_prompt(error)
    
    # 3. LLM 自主调用工具分析
    #    - 可能先 read_lines 看上下文
    #    - 可能 search_text 搜角色名确认是不是对话
    #    - 最终调用 apply_fix 或 skip_error
    
    # 4. LLM 提交修正 → 自动调用 Verifier 确认
    fix_result = tools.apply_fix(...)
    verdict = verifier.verify(error, fix_result)
    
    if verdict == "pass":
        error_queue.mark_fixed(error_id)
        save_checkpoint()  # 每修一处就保存
    elif verdict == "uncertain":
        # 低置信度，记录到日志，人工后续审核
        error_queue.mark_skipped(error_id, "uncertain")
    else:  # fail
        tools.revert_fix(error_id)
        error_queue.mark_failed(error_id)
        # 可以尝试其他策略后重试
```

#### System Prompt 设计

```
你是一个小说文本纠错专家。你的任务是对小说中人物对话的包裹符号进行纠错。

## 核心规则
1. 小说人物对话的唯一合法包裹符号是成对的全角直角引号「」
2. 你需要修复以下类型的错误：
   a. 「「 或 」」连续出现（其中一个符号应为其他符号）
   b. [ ]、【】等非「」符号包裹了对话
   c. 对话内部缺失「或」，导致对话与旁白被错误合并
   d. 成对不匹配（有「无」或有」无「）

## 你的职责范围（严格遵守）
1. ✅ 你的任务是修正——不是发现错误
2. ✅ 错误位置已经由规则检测器定位好了，你只需要调用 get_next_error() 获取
3. ❌ 绝对禁止自己去找小说里还有没有其他错误
4. ❌ 绝对禁止一次性处理多个错误
5. ❌ 绝对禁止修改除了指定错误位置之外的其他内容

## 可用工具
- read_lines(start, end): 读取原文指定行范围
- read_offset(offset, context): 以指定位置为中心读取上下文
- search_text(keyword): 搜索关键词
- get_next_error(): 获取下一个待处理错误
- apply_fix(error_id, start, end, replacement): 执行修正（每次只修一个）
- skip_error(error_id, reason): 标记跳过
- revert_fix(error_id): 回滚上一次修正

## 纠错原则
1. 只修正符号错误，不修改任何原文内容
2. 遇到不确定的情况，先用工具读更多上下文，不要猜
3. 读完了还是不确定 → 调用 skip_error，不要强修
4. 每次修正后会自动验证，验证失败会自动回滚，不用担心改坏
5. 修完一个就调用 get_next_error() 获取下一个，不要自己乱找
```

## 八、与 v1 的关键区别

| 维度 | v1 | v2 |
|------|----|----|
| **定位错误** | 滑动窗口无差别扫描 | 规则检测器精确定位 |
| **LLM 角色** | 被动分类器（看固定窗口） | 主动代理（自主阅读） |
| **LLM 任务范围** | 同时检测 + 修正 | 只修正，不检测（检测全由规则完成） |
| **错误处理方式** | 一次处理一批错误 | 一次只处理一个错误，修好再修下一个 |
| **上下文获取** | Python 预切 800 字符 | LLM 调用 read_lines 自取 |
| **修正方式** | 整段替换 + 相似度校验 | 精确偏移替换 + Verifier 确认 |
| **模型调用** | 每次 4 个模型做共识 | 1 个模型 + Verifier 校验 |
| **断点续跑** | 不支持 | checkpoint 持久化 |
| **修改报告** | 无 | JSONL 详细记录 |

  ```python
      text = TextLoader().load(novel_path)
      while True:
          # Phase 1: 全量检测
          errors = DetectorPipeline().run(text)
          if errors.total == 0:

          agent = CorrectionAgent(text, errors, model_client, verifier)
          agent.run()
          # Phase 3: 重新检测
          second_pass = False  # 重置二次确认标志
  ```

## 十、关键风险与控制

| 风险 | 缓解措施 |
|------|----------|
| LLM 过度修正（把不是错误的也改了） | Verifier 验证 + 严格只在替换范围内修改 |
| LLM 陷入循环（反复修正同一个位置） | 轮次上限 + 连续 fail N 次后标记跳过 |
| 规则检测器误报（把正常文本当成错误） | ErrorQueue 允许 skip，由 LLM 判断 |
| 超长文本处理慢 | 按错误逐个处理，每修一处 checkpoint |
| Ollama tool calling 不稳定（同 dialoop 经验） | 两级协议：原生 tools → JSON action 兜底 |

## 十一、与 opencode-novel-loop 的集成路径

novel_correct_v2 的输出 = opencode-novel-loop 的输入：

```
novel_correct_v2:  raw_novel.txt → corrected_novel.txt
                                ↓
opencode-novel-loop:  corrected_novel.txt → labeled.txt
                                ↓
novel-voice-cast:  labeled.txt → 有声书
```

**接口约定**：
- 输入：任意 txt 文件（UTF-8/UTF-16/GBK 自动检测）
- 输出：UTF-8 编码，`「」` 符号统一，对话格式规范
- 行号保持稳定（如果有行号变化，在报告中记录偏移量映射）
