# novel_correct_v2 — 项目全景与背景

> 本文档梳理整个小说转语音大项目的全貌、各子项目的现状与依赖关系，作为 novel_correct_v2 开发的「前情提要」。

---

## 一、终极目标

**输入小说 → AI 纠错 → AI 标注说话人 → 用户上传角色音频 → 音色克隆 + TTS → 输出有声书**

轻小说的对话通常不标注说话人，只用 `「」` 包裹。市面上的 TTS 工具要么只有单一朗读者，要么配置复杂。因此我想做一个完整的管线，从原始文本一路走到带角色演绎的有声书。

## 三、详细背景

### 3.1 novel_correct v1（GitHub）

> 链接：https://github.com/caiyilian/novel_correct

OCR 识别小说文本时，常出现以下问题：
- 对话符号 `「」` 被误识别为 `[` `[`、`［］`、`【】` 等
- 其他标点混乱：全半角混用、省略号乱码、破折号断裂
- 错别字、形近字错误

v1 版本采用 Python + Ollama 本地模型进行纠错，已经给出了基础版本。v2 目标是进一步提高准确率。

### 3.2 说话人标注工具的演进：v1 → v2 失败方案 → v2 最终方案

此项目经历了两次重大方案迭代，理解这个演进过程对 novel_correct_v2 的设计很有参考价值。

---

#### 第一阶段：novelSpeakLabel（v1）

> GitHub: [caiyilian/novelSpeakLabel](https://github.com/caiyilian/novelSpeakLabel)

最简单的方案：Python + Ollama，将文本片段直接丢给 LLM，让模型一次性判断说话人。

**问题**：效果不好。
- LLM 只能看到 Python 预先切好的固定文本块
- 无法主动翻阅上下文寻找线索
- 遇到不确定的情况只能猜或标 unknown
- 准确率不足以支撑后续 TTS 管线

---

#### 第二阶段：speaker_labeling_plan_v2.md（V2 方案 — 失败）

> 路径：`E:\projects\opencode-novel-loop\speaker_labeling_plan_v2.md`

这个方案试图用 Python 精细控制整个流程，设计了复杂的流水线架构：

| 组件 | 职责 |
|------|------|
| Chunk 切分 | 按 `max_paragraphs_per_chunk: 12-20` 将小说分成块 |
| 固定上下文窗口 | lookback 2-4 段，lookahead 0-2 段 |
| Task A | 当前 chunk 初标注 |
| Task B | 当前 chunk 短期摘要 |
| Task C | 长期摘要更新 |
| Task D | 新增实体发现 |
| Task E | 已有角色卡片更新 |
| Task F | 当前 chunk 回看补标 |
| 三层记忆 | 短期摘要、长期摘要、角色卡片 |
| 实体分类 | character / mystery / npc_group |

**失败原因**（架构级硬伤）：

```
V2（被动分类器）：
  Python 切好小块上下文 → 塞给 LLM → LLM 在小块里找
  → 不够 → 标 unknown

OpenCode（主动代理）：
  LLM 收到任务 → 自己决定读哪里 → 不够 → 再读
  → 还不够 → 跳到前面章节 → 够了 → 输出
```

关键对比：

| 维度 | V2 方案 | 正确的做法 |
|------|---------|-----------|
| 谁控制阅读 | Python 预切分固定窗口 | LLM 自主导航 |
| 上下文大小 | 固定（lookback/lookahead） | 动态，LLM 自己决定 |
| 跨章节关联 | 仅靠摘要（会漂移） | 直接翻原文 |
| 不确定时 | 标 unknown 放弃 | 继续读更多 |
| 错误传播 | 摘要漂移，Task A→B→C 层层放大 | 每次直接读原文 |
| 架构本质 | 被动分类器 | 主动代理 |

具体问题：
1. **固定窗口漏线索**：说话人线索在第 6 段前，但只给了前 3 段 lookback
2. **摘要会漂移**：错误层层传播（Task A→B→C→D→E→F）
3. **LLM 被动接受**：无法主动要求更多上下文
4. **记忆不如原文**：摘要一定会丢失信息
5. **系统复杂但效果受限于窗口瓶颈**

#### 第三阶段：最终方案 — 独立 Python Agent（当前 v2 实现）

> 路径：`E:\projects\opencode-novel-loop\dialoop/`

**核心洞察**：放弃预设上下文窗口，给 LLM 工具，让它自己翻书。

##### 设计原则
1. **LLM 是主动代理，不是被动分类器** — 给它读小说的工具，让它自己决定看哪里
2. **工具调用（Tool Calling）** — 定义 `read_novel`、`search_novel`、`get_next_dialogue`、`submit_labels` 等工具
3. **增量标注** — 按 `labeled.txt` 行数断点续跑
4. **模型无关** — 通过 OpenAI-compatible API 支持任何模型（Ollama、DeepSeek、Qwen 等）
5. **质量校验环** — 每次提交都经过验证，不通过则重试或降级

##### 核心架构

```
用户
  |
  | dialoop novel.txt --model qwen3:32b
  v
CLI 层
  |
  | 构建运行配置
  v
Runner / Orchestrator
  |
  | 管理进度、重试、最大轮数、日志
  v
Agent Loop
  |
  | 给模型提供工具：
  |   read_novel(start, end)      → 读取指定行范围
  |   get_next_dialogue(batch)    → 获取下一批待标注对话
  |   submit_labels(speakers)     → 写入 labeled.txt
  |   search_novel(keyword)       → 搜索关键词/角色名
  |   get_progress()             → 查看当前进度
  |   read_active_context()      → 读取当前对话的上下文
  |   locate_identity()          → 定位角色身份
  |   resolve_identity()         → 解析角色身份
  |   normalize_speaker()        → 归一化角色名
  v
LLM（模型）: 自主决定读哪里、搜什么、何时提交
```

##### 演进中保留的高级特性（阶段 5-6）

通过多阶段迭代，在基础 Agent Loop 之上逐步叠加了以下能力：

| 阶段 | 特性 | 说明 |
|------|------|------|
| 5A | 评估基线 | 建立标注错误分类体系 |
| 5B | 证据化标注 | 标注时要求附带行号证据 |
| 5C | 风险门控 | 低置信度的提交自动驳回 |
| 5D | 主 agent + 辅助工具分工 | 避免主 agent 分心 |
| 5E | 身份后置查找 agent | 专门定位陌生角色名 |
| 5F | 角色库与显示名归一 | 防止「赫萝」「赫罗」「贤狼」混用 |
| 6-1 | Coordinator 调度 | 多 agent 编排框架 |
| 6-2 | Verifier + Arbiter | 校验 + 仲裁闭环 |
| 6-3 | 独立 Identity Locator/Resolver | 专门用于角色搜索的 agent |
| 6-4 | Name Normalizer | 角色名归一化 |

##### 对话提取方式

```python
DIALOGUE_PATTERN = re.compile(r"「([^」]+)」")
```

按行扫描，匹配所有 `「」` 包裹的内容，记录行号和文本。
**如果 OCR 把 `「」` 识别成了其他符号，整个提取会完全失败。** 这就是纠错必须排在第一位的根本原因。

##### 标注流程

1. `DialogueIndex` 从文本中提取所有 `「」` 对话
2. 批量取出待标注对话（附上下文）
3. Agent 调用工具链自主搜索、阅读、定位角色
4. `Coordinator` 协调多个 Agent 的决策
5. `Verifier` 校验标注质量，必要时 `Arbiter` 仲裁
6. 结果追加写入 `labeled.txt`

### 3.3 novel-voice-cast — TTS 有声书合成

> 路径：`E:\projects\novel-voice-cast`

#### 技术栈

| 层 | 选型 |
|------|------|
| 后端 | FastAPI + SQLite + SQLAlchemy |
| 前端 | Vue 3 + Vite + Naive UI |
| TTS | edge-tts（预设）+ 音色克隆（VoxCPM，计划接入 IndexTTS2） |
| 音频处理 | pydub + soundfile |

#### 流程
```
labeled_novel.txt ──→ 解析提取对话+角色 ──→ Web UI 分配音频 ──→ 音色克隆+TTS ──→ 输出完整有声书
```

#### 核心模块
- `parser.py` — 解析已标注文本，支持 `「」` 格式、章节标题、说话人前缀格式
- `ollama_client.py` — Ollama API 封装（支持 tool calls、重试机制）
- `chapter_extractor.py` — 章节提取（规则 + LLM 双重策略）
- `gender_identifier.py` — 角色性别自动识别
- `emotion_labeler.py` — 情感标注引擎
- `tts/` — 多 TTS 引擎抽象层（PresetProvider、VoiceCloneProvider）

#### parser.py 的对话格式支持

```python
# 「对话」——说话人       （日式标注：对话在前，说话人在后）
# 说话人：「对话」         （中式标注：说话人在前，对话在后）
# 独立「对话」             （无说话人标注，由标注阶段补充）
```
三种格式全部兼容。但**所有格式都依赖 `「」` 符号作为对话标记**。

## 四、依赖链与关键风险

```
                    ┌─────────────────────┐
                    │  OCR 原始文本         │
                    │  (含大量识别错误)      │
                    └────────┬────────────┘
                             │
                             ▼
                    ┌─────────────────────┐
                    │  (1) novel_correct   │  ← ⚠️ 当前要做的
                    │  OCR 纠错            │
                    │  修复「」符号等       │
                    └────────┬────────────┘
                             │
                             ▼
                    ┌─────────────────────┐
                    │  (2) opencode-novel  │
                    │  -loop              │
                    │  说话人标注          │
                    └────────┬────────────┘
                             │
                             ▼
                    ┌─────────────────────┐
                    │  (3) novel-voice-    │
                    │  cast               │
                    │  TTS 有声书合成      │
                    └─────────────────────┘
```

**关键风险：**

| 风险 | 影响 |
|------|------|
| `「」` 被 OCR 识别为 `[]` `【】` 等 | opencode-novel-loop 的 `DialogueIndex` 无法提取对话 → 标注失败 |
| 对话中混入 OCR 乱码 | Agent 推理说话人时被干扰 → 标注准确率下降 |
| 标点符号混乱（全半角混用） | parser 解析出错 → 对话/叙述边界划分错误 |
| 章节标题格式不统一 | 章节提取失败 → 有声书分段错乱 |

**结论：novel_correct_v2 是整个管线的第一道关卡，其纠错质量决定了后续所有步骤的效果上限。**

## 五、设计原则（跨项目复用）

从 opencode-novel-loop 和 novel-voice-cast 中提炼的通用设计原则：

1. **Agent + 工具调用模式**：让 LLM 通过工具操作本地数据，而非在上下文里塞入全文
2. **分层纠错**：确定性规则先做粗处理（如全半角统一），LLM 再做语义级精细化调整
3. **增量处理**：支持中断恢复，标注/纠错进度可持久化
4. **模型无关**：通过 Ollama 兼容 API，可切换任意本地模型
5. **质量校验环**：每次 LLM 输出后校验，不通过则重试或降级
