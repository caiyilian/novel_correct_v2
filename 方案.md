- 不以“精确对话匹配率”作为完成条件，因为当前差异主要来自 OCR 文本内容，不属于本工具链范围。

### 1.3 允许修改的范围

后续实施阶段允许修改：

- 对话包裹符号：`[]`、`【】`、`［］`、`{}`、`“”`、`""` 等到 `「」` 的归一化。
- 成对关系：补齐缺失的 `「` 或 `」`，删除或翻转明显多余的引号。
- 标点风格：半角 `?`、`!`、`~`、`.` 等在中文语境下的全角化或句末标点判断。
- 空 `「」` 清理。

除上述范围外的正文差异只记录，不作为本阶段修复对象。

## 3. 总体策略

### 3.1 第 3 卷采用单独前置阶段

第 3 卷不是普通的零散 OCR 错符号，而是全卷格式体系不同：原文主体使用弯双引号 `“”`，没有 `「」`。因此不建议直接把 2,000 多个弯引号全部丢给 `WrongSymbolDetector` 和普通候选队列逐条处理。

推荐策略：

1. 先做第 3 卷专属的“弯引号前置归一化”阶段，将可靠成对的 `“...”` 转为 `「...」`。
2. 再运行现有检测器处理剩余异常：不成对弯引号、漏包裹、混入 `{}` 等。
3. 最后使用答案文件只做结构验收和缺口定位，不复制答案正文。

这样能把系统性格式差异和真正的异常分开，降低误修风险，也能避免普通检测队列被第 3 卷淹没。

### 3.2 分阶段并行，关键卷串行收敛

执行方式采用“分阶段并行、关键卷串行”：

- 基准统计、低风险标点、最终验证可以对所有卷批量运行。
- 第 3 卷和第 2 卷需要单独串行收敛，先解决最大结构风险。
- 第 4、6 卷作为不配平组处理。
- 第 5、7、8、9、10 卷作为小缺口组处理。

推荐处理顺序：

1. 冻结第 1 卷。
2. 先跑 Stage 0~2，统一基准、产物规范和低风险预处理。
3. 只读审计阶段可以并行推进：Stage 3a、Stage 4a、Stage 6a、Stage 7a 不修改正文，可以同时产出候选和缺口报告。
4. 写入修复阶段必须按卷串行：先完成第 3 卷前置归一化和结构收敛，再完成第 2 卷 `[]`、不配平和半角句号专项。
5. 处理第 4、6 卷不配平。
6. 处理第 5、7、8、9、10 卷少量缺口。
7. 最后跑 10 卷统一验收。

这个顺序的关键点是：读报告可以并行，写正文不要跨卷混批；同一卷永远只有一个 canonical corrected 文件在推进，避免 offset、报告和白名单互相错位。

### 3.3 答案文件使用原则

`data/answer/answer_第X卷.txt` 的用途限定为：

- 统计目标 `「」` 数量。
- 进行结构对齐，定位“少了几段”“多了几段”“从哪一段开始错位”。
- 验收最终对话包裹结构。

禁止用途：

- 直接复制答案正文覆盖 OCR 原文。
- 用答案内容替换 OCR 字词。
- 让 LLM 根据答案改写正文。

### 3.4 Ollama 使用边界

Ollama 可以继续沿用第 1 卷后期的“微案例裁决器”角色，但不能接管全文修正。

适合使用 Ollama 的场景：

- 高风险标点候选裁决，例如 `.` 在中文上下文中应保留、改 `。`、改 `？`、改 `！`，或判断为省略号残片。
- 第 3 卷弯引号归一化后的少量异常 case，例如跨段弯引号、不成对弯引号、引号内是否像对话。
- 第 2 卷 `[]` 残留的上下文分类，例如对话包裹、注释、编号、旁注。
- 小缺口卷中 1~5 个疑似漏包裹位置的二审，前提是 Python 已给出候选 offset 和候选替换。

不适合使用 Ollama 的场景：

- 全文重写。
- 根据答案正文改 OCR 字词。
- 自行生成 offset 或自由插入 `「」`。
- 批量应用没有经过 Python verifier 的修改。

约束要求：

- Ollama 输入必须是一条候选或一个很小的候选组，包含上下文、候选 id、可选动作和禁止事项。
- Ollama 输出必须是单行 JSON，且只能选择 `apply` / `keep` / `uncertain`。
- Python 必须校验 `decision`、`candidate_id`、offset、原文片段和修后硬指标。
- 每批 Ollama 决策都必须输出 `decisions_第X卷.jsonl`，记录 token usage、模型名、case id、候选来源和验证结果。

### 3.5 批次大小与关键卷顺序决策

“每批最多 50 条”应作为硬上限，而不是默认目标。实际批次大小按风险和同质性调整：

- 校准批次：每种新候选类型先跑 10~20 条，用来验证 prompt、JSON 解析、offset guard 和 verifier 是否稳定。
- 同质候选：第 2 卷半角句号这类结构一致的候选，可以按 40~50 条成批处理。
- 异质候选：第 3 卷弯引号异常、第 2 卷 `[]` 高风险、跨卷混合候选，建议 20~30 条一批，最多不超过 50 条。
- 失败收缩：任一批出现 verifier 失败、offset guard 失败，或 `uncertain` 比例超过 30%，下一批必须缩小到 10~20 条，并先复盘分类规则。
- 不混风险：同一批尽量只包含同一卷、同一风险类型、同一候选生成规则。

第 2 卷和第 3 卷的顺序按“可并行审计、串行写入”执行：

1. Stage 3a 与 Stage 4a 都是只读审计，可以并行，尽快暴露第 3 卷异常规模和第 2 卷 `[]` 分类分布。
2. Stage 3b~3d 写入修复优先于第 2 卷写入修复，因为第 3 卷是全卷格式体系差异，完成后能先验证前置归一化路线是否成立。
3. Stage 4b~4e 紧随其后，先修 `[]` 和不配平，再处理 155 个半角句号；半角句号不应早于 bracket 结构修复批量应用。
4. 如果 Stage 4a 发现第 2 卷 `[]` 低风险候选极少，可以在 Stage 3c 等待人工复核时穿插 Stage 4b，但不能让两个写入阶段同时修改同一个卷。

## 4. 阶段划分

### Stage 0 — 基准冻结与数据契约

输入：

- `tools/batch_run_v2_10.py`

输出：

- `output/batch_report_v2_10.json`
- `output/batch_report_v2_10.txt`
- 卷级目标表，记录每卷目标 `「/」` 数量和当前缺口。

验收标准：

- 10 个原始文件和 10 个答案文件全部存在。
- 答案文件合计 `「/」` 为 12,776/12,776。
- 第 2~10 卷答案合计 `「/」` 为 11,427/11,427。
- 每卷当前状态和 Issue #85 的基准表一致，若脚本输出与 Issue 表有差异，必须在报告中标出并人工确认采用哪一个基准。

### Stage 1 — 产物与日志规范

目标：先统一后续阶段的产物命名，保证每个修正都可追溯。

输入：

- Stage 0 的批量报告。
- 当前 `output/corrected_第X卷.txt` 产物约定。

输出：

- 每卷一个最终文本：`output/corrected_第X卷.txt`。
- 每卷一个结构报告：`output/verify_第X卷.json`。
- 每卷一个对话对齐报告：`output/compare_第X卷.json`。
- 每卷一个人工或 LLM 决策日志：`output/decisions_第X卷.jsonl`。
- 一个最终汇总：`output/final_report_10_volumes.json`。

验收标准：

- 每条自动或人工修正至少记录：卷号、offset、原片段、替换片段、候选来源、决策方式、验证结果。
- 每条跳过项必须记录原因，不能只有 `skipped` 状态。
- 最终报告能追溯每卷剩余异常、白名单异常和已完成项。

### Stage 2 — 全卷低风险预处理

目标：先消除确定性、低风险的格式问题，为高风险修复减少噪声。

输入：

- 原始 10 卷文本。
- `tools/clean_empty_dialogues.py`
- `tools/generate_style_candidates.py`

处理原则：

- 空 `「」` 删除只在确定为空包裹时执行。
- 低风险标点自动应用，例如中文上下文中的 `?`、`!`、`~`。
- 半角句号 `.` 默认高风险，不自动应用。

输出：

- 每卷预处理后的 `output/corrected_第X卷.txt`。
- 每卷 `style_candidates_第X卷.jsonl`。
- 高风险候选单独输出到 `style_candidates_第X卷_high.jsonl`。

验收标准：

- 第 2~10 卷低风险标点自动修正数量应与基准约 1,267 接近。
- 自动处理不得让任一卷的 `「/」` 差值扩大。
- 自动处理后重新运行 `main.py --detect`，检测错误总数不得高于 Stage 0 基准。

### Stage 3a — 第 3 卷弯引号基线审计

目标：先只统计和分类第 3 卷的弯引号，不改正文。

输入：

- `data/ori_story/第3卷.txt`
- `data/answer/answer_第3卷.txt`
- Stage 2 的第 3 卷预处理结果。

输出：

- `output/vol3_curly_quote_audit.json`
- 第 3 卷弯引号候选清单，按可靠成对、跨段、不成对、疑似非对话分组。
- 第 3 卷当前答案目标段数和预计缺口说明。

验收标准：

- 弯引号数量、成对候选数量、异常候选数量全部写入报告。
- 报告不修改 `output/corrected_第3卷.txt`。
- 明确哪些候选可以规则批量转，哪些必须进入 Stage 3c 二审。

### Stage 3b — 第 3 卷可靠弯引号规则转换

目标：只把 Stage 3a 中“可靠成对、同段、明显对话”的弯引号转换为 `「」`。

输入：

- Stage 3a 的审计报告。
- Stage 2 的第 3 卷预处理结果。

输出：

- `output/corrected_第3卷_stage3b.txt`
- `output/vol3_curly_quote_apply_report.json`

验收标准：

- 只应用 Stage 3a 标记为 low-risk 的候选。
- 转换前后除包裹符号外，正文字符不得变化。
- 转换后 `「` 和 `」` 必须配平；如不配平，整阶段回滚。
- 剩余异常候选进入 Stage 3c，不在本阶段强行处理。

### Stage 3c — 第 3 卷弯引号异常小批量裁决

目标：处理 Stage 3a/3b 留下的少量异常弯引号候选。

输入：

- `output/corrected_第3卷_stage3b.txt`
- Stage 3a 的异常候选清单。

处理原则：

- 首批建议 10~20 条；后续按异常类型分批，最多 50 条候选，避免一次阶段过大。
- 可使用 Ollama，但只能在 Python 给出的候选动作中选择。
- `uncertain` 或 verifier 失败的候选进入人工审核，不得自动应用。

输出：

- `output/decisions_第3卷_stage3c.jsonl`
- `output/corrected_第3卷_stage3c.txt`
- `output/verify_第3卷_stage3c.json`

验收标准：

- 本批所有候选均有 `apply` / `keep` / `uncertain` 记录。
- 每个 `apply` 均通过 quote balance、offset guard、非标准符号检查。
- 本阶段结束后明确剩余候选数；若仍有剩余，继续开 Stage 3c-2，而不是扩大单阶段范围。

### Stage 3d — 第 3 卷结构收敛验收

目标：确认第 3 卷是否已达到答案结构目标，或只剩可解释白名单。

输出：

- `output/corrected_第3卷.txt`
- `output/decisions_第3卷.jsonl`
- `output/compare_第3卷.json`
- `output/verify_第3卷.json`

验收标准：

- 第 3 卷最终 `「/」` 为 1215/1215。
- 原文中的弯双引号不再作为对话包裹符残留。
- `compare_dialogues` 能提取出 1215 段，且无空对话。
- `main.py data/ori_story/第3卷.txt --detect` 的结构类问题全部关闭，或全部进入明确白名单。
- 若仍未达到 1215 段，不关闭第 3 卷，另开下一小阶段定位缺口。

### Stage 4a — 第 2 卷 `[]` 和不配平定位

目标：先定位第 2 卷不配平和 `[]` 残留来源，不处理 155 个半角句号。

输入：

- `data/ori_story/第2卷.txt`
- `data/answer/answer_第2卷.txt`

输出：

- `output/vol2_bracket_audit.json`
- `output/vol2_bracket_candidates.jsonl`

验收标准：

- 144 个 `[]` 残留均被分类：对话包裹、注释、编号、旁注、uncertain。
- 不修改正文。
- 明确哪些候选可规则修复，哪些需要 Ollama/人工裁决。

### Stage 4b — 第 2 卷 `[]` 低风险规则修复

目标：只处理 Stage 4a 中确定属于对话包裹的低风险 `[]` 候选。

- quote balance 相比输入变好或不变，不得变差。
- 非对话用途 `[]` 不在本阶段处理。
- 本阶段结束后剩余 `[]` 数量和原因写入报告。

### Stage 4c — 第 2 卷 `[]` 高风险裁决

目标：处理 Stage 4b 后剩余的 `[]` 高风险候选。

处理原则：

- 可使用 Ollama 判断候选类型，但只能返回 `apply` / `keep` / `uncertain`。
- 每批建议 20~30 条，最多 50 条候选；第 2 卷 `[]` 如超过 50 条，拆 Stage 4c-1、4c-2、4c-3。
- `keep` 必须说明非对话用途；`uncertain` 进入人工审核。

输出：

- `output/decisions_第2卷_brackets.jsonl`
- `output/corrected_第2卷_stage4c.txt`
- `output/verify_第2卷_stage4c.json`

验收标准：

- 本批候选全部有决策记录。
- 每个 `apply` 通过 verifier。

## 6. 关键风险与控制

| 风险 | 表现 | 控制方式 |
|------|------|----------|
| 把答案正文复制进修正文本 | OCR 内容被非范围内改写 | 明确禁止；所有 diff 只允许符号和标点范围 |
| 第 3 卷弯引号批量误转 | 注释或非对话引号被包为 `「」` | 前置归一后必须跑结构对齐和人工抽检 |
| 第 2 卷半角句号误判 | `.` 可能是句号、问号、感叹号或省略号 | 155 条逐条裁决，不自动应用 |
| Ollama 过度改写 | 模型根据上下文自由改正文 | 只允许 JSON 裁决候选，Python 校验并应用 offset patch |
| 阶段过大难以回滚 | 一次处理多卷或数百候选导致定位困难 | 高风险候选每批不超过 50 条，小缺口卷逐卷验收 |
| `《》` 被误判为对话符号 | 书名号在答案中可能合法存在 | 建立白名单，按非对话用途关闭 |
| 配平但结构仍错 | `「」` 数量一致但段落错位 | `compare_dialogues` 段数和错位报告必须纳入验收 |
| 检测器误报导致无法零错误 | 合法符号被检测为 wrong_symbol | 用白名单关闭，但 P0/P1 结构错误不能随意白名单 |

## 7. Issue 讨论问题的结论

1. 整体分多少阶段？

   仍按 Stage 0 到 Stage 10 组织，但把工作量大的阶段拆成 a/b/c 子阶段。执行时每个子阶段都应单独提 issue、单独 PR、单独验收，尤其是第 3 卷、第 2 卷和跨卷高风险候选。这样可以保持每次变更小、可回滚、可复核。

2. 第 3 卷弯引号应在检测器层解决还是前置阶段解决？

   推荐单独前置阶段解决。第 3 卷是全卷格式差异，不是零散错误；前置归一化后再让检测器处理残留异常，风险更低。

3. 第 2~10 卷逐卷串行还是分阶段并行？

   采用分阶段并行、关键卷串行。批量统计、低风险预处理和候选定位可以并行；第 3 卷、第 2 卷、第 4/6 卷结构修复应按优先级串行收敛；小缺口卷先统一定位，再按卷逐个修复和验收。

4. 哪些阶段复用第 1 卷工具链？

   复用 `TextLoader`、`DetectorPipeline`、`clean_empty_dialogues`、`generate_style_candidates`、`compare_dialogues`、`verify_against_answer`、`batch_run_v2_10.py`、`polish_judge`、`run_polish_judge.py`、`apply_polish_decisions.py` 和 `main.py --detect`。第 3 卷前置归一化、缺口定位、高风险批次拆分和最终 10 卷汇总建议新增工具，但不在本 Issue 内实现。

5. 如何定义“全部完成”？

   全部完成等于：10 卷最终 `「/」` 合计 12,776/12,776；每卷段数等于答案目标；无空对话；无未解释的非标准对话包裹符；无未关闭 P0/P1 结构错误；所有高风险候选和白名单项均有可追溯记录。
