## Stage 16 — 扩展候选生成器覆盖率（✅ 已完成）

### 结果

- 默认规则预检：52 fixed, 689 skipped, 0 failed
- `--llm-decision-fallback`：**171 fixed, 0 skipped, 0 failed**（第 1 卷全部修正）
- PR #38 → merge 到 master（dc07046） ✅

### 现状

第 1 卷 171 个错误中：
- 7 个修正（wrong_symbol ×1, consecutive ×2, unpaired ×3, long_dialogue ×1）
- **163 个跳过**，原因是候选生成器无法为这些错误生成候选

### 要解决的问题

`src/agent/candidates.py` 中的各类型生成器覆盖度不足：

| 错误类型 | 当前候选生成逻辑 | 漏掉的场景 |
|---------|----------------|-----------|
| `consecutive` | 只覆盖 `」\n「` 等少数模式 | 跨行连续、混合空白字符、非标准符号导致的连续等 |
| `long_dialogue` | 只覆盖句号+叙述词拆分 | 问号/感叹号/省略号结尾的对话段 |
| `unpaired` | 补左/补右/删除 | 更多复杂非配对情况 |
| `wrong_symbol` | 单字符替换 + 成对替换 | 嵌套符号、半角括号变体等 |

### 任务

1. 梳理第 1 卷 163 个跳过的错误，分类统计
2. 对每一类扩展候选生成逻辑
3. 用回归测试验证：之前跳过的错误中，新增候选后能正确修正的比例
4. 迭代直到覆盖率达到 80%+

### 验收标准

- 第 1 卷 fixed 数量从 7 提升到 30+
- 候选生成器对 consecutive 的覆盖率达到 80%+
- 所有扩展的候选逻辑都有对应单元测试

## Stage 18 — 候选生成器再扩展（✅ 已完成）

### 结果

- `tests/test_stage18.py` — **9 passed, 0 failed**
- 修正后文本候选覆盖率：**129/130 (99%)**
  - consecutive: 56/56 (100%)
  - wrong_symbol: 62/62 (100%)
  - long_dialogue: 11/12 (92%)
- Stage 16 回归：6/6 passed
- Stage 17 回归：6/6 passed
- 纯规则多轮收敛：130→69→60→59（54% reduction, monotonic decrease）
- `--llm-decision-fallback` 模式预期可在 ≤3 轮收敛到 0（需 LLM 实际运行验证）

### 解决的关键问题

1. **`ErrorRecord.offset` 指向空白/换行**：`_resolve_offset()` / `_scan_to_char()` / `_scan_to_bracket()` 向前扫描 30 字符定位实际符号
2. **修正后文本 130 个新错误的候选覆盖**：大部分修正后错误是新的 `wrong_symbol`（62 个）和 `consecutive`（56 个），候选生成器已全部覆盖
3. **`_consecutive()` 跨行连续**：增加了跨行 Candidate 3（翻转换行前括弧）
4. **`_long_dialogue()` 拆分增强**：Strategy 3（`？/！/……` + 叙述词拆分）、Strategy 4（`\n\n` + 叙述前插入 `」`）

### 尚未解决的问题

- 1 个 `long_dialogue`（offset ~100082）无有效拆分点，无法生成候选
- 纯规则模式 3 轮后残留 59 个错误（需 `--llm-decision-fallback` 处理）
