### 1.4 关键发现——检测器覆盖度盲区

Stage 18 宣称 151 错误 100% 覆盖，但对比答案 `data/answer_第1卷.txt`（由 novelSpeakerV4 人工整理，1349 对「」完美平衡），发现严重差距：

| 」 | 1163 | **1283** | **1349** |

| 不成对 | 42（「多） | 25（」多） | **0** |

**核心问题**：修正后文件中仍有 **158 个非标准符号**未被检测器发现。

### 1.3 存在的问题（续）

| # | 问题 | 具体表现 |
|---|------|----------|
| 0 | **检测器覆盖度盲区** | 修正后仍有 158 个非标准符号残留（65[, 93], 4弯引号），未被任何检测器捕获 |

| 6 | **无结构化日志** | 只看 correction_report.json 不能分析趋势、瓶颈、回归 |

## 二、改进方向

### 方向 A — Token 跟踪与上下文监控

**目标**：每次 LLM 调用都记录 token 消耗，监控上下文窗口利用率。

#### 当前问题

`ChatResult` 虽然存了 `raw` 字段（完整 payload），但：
1. 没有提取 `usage.prompt_tokens` / `usage.completion_tokens`
2. 没有暴露 `prompt_eval_count` / `eval_count`（Ollama API 返回的字段名）
3. 没有任何地方记录或汇总 token 消耗
4. 没有任何地方计算上下文窗口使用率

#### 参考 novelSpeakerV4

```python
# novelSpeakerV4 的 token 跟踪模式：
text, pec, ec, tool_calls = call_ollama(...)  # 每个 LLM 调用都返回 token 数
total_tokens = pec + ec                        # 立即累加
log_agent(..., pec=pec, ec=ec, total_tokens=...)  # 记录到每轮日志

# 上下文监控：
TOKEN_BUDGET = 18000  # 预算控制
if cumulative_tokens >= TOKEN_BUDGET:
    print(f"Token budget reached ({cumulative_tokens} >= {TOKEN_BUDGET}), stopping")
```

#### 改进方案

**1. 增强 ChatResult，附带 token 信息**

```python
# src/model/client.py
@dataclass(frozen=True)
class TokenUsage:
    prompt_tokens: int = 0
    completion_tokens: int = 0
    total_tokens: int = 0

@dataclass(frozen=True)
class ChatResult:
    content: str = ""
    tool_calls: list[ToolCall] = field(default_factory=list)
    raw: dict[str, Any] = field(default_factory=dict)
    usage: TokenUsage = field(default_factory=TokenUsage)  # 新增
```

**2. 在 _parse_chat_result 中解析 usage**

Ollama API 返回两种格式，都需要兼容：

```python
# Ollama /api/chat 格式
{
  "prompt_eval_count": 123,
  "eval_count": 456
}

# OpenAI-compatible /v1/chat/completions 格式
{
  "usage": {
    "prompt_tokens": 123,
    "completion_tokens": 456,
    "total_tokens": 579
  }
}
```

增强 `_parse_chat_result`：
```python
def _parse_chat_result(self, payload: dict) -> ChatResult:
    # ... 现有代码解析 content、tool_calls ...
    
    # 新增：提取 token 使用
    usage = TokenUsage()
    # Ollama 原生格式
    pec = payload.get("prompt_eval_count", 0)
    ec = payload.get("eval_count", 0)
    if pec or ec:
        usage = TokenUsage(prompt_tokens=pec, completion_tokens=ec, total_tokens=pec+ec)
    else:
        # OpenAI 兼容格式
        u = payload.get("usage", {}) or {}
        usage = TokenUsage(
            prompt_tokens=u.get("prompt_tokens", 0),
            completion_tokens=u.get("completion_tokens", 0),
            total_tokens=u.get("total_tokens", 0),
        )
    
    return ChatResult(content=..., tool_calls=..., raw=payload, usage=usage)
```

**3. 创建 TokenTracker 类**

```python
# src/model/token_tracker.py
@dataclass
class TokenRecord:
    timestamp: float
    source: str           # 哪个 agent/环节
    error_id: str         # 关联的错误 ID
    error_type: str       # 错误类型
    prompt_tokens: int
    completion_tokens: int
    total_tokens: int
    context_window_pct: float  # 上下文窗口使用率

class TokenTracker:
    """全局 token 消耗追踪器"""
    
    def __init__(self, context_limit: int = 40960):
        self._records: list[TokenRecord] = []
        self._context_limit = context_limit
        self._round_totals: dict[int, int] = {}  # round → total_tokens
    
    def record(self, source: str, error_id: str, error_type: str,
               usage: TokenUsage) -> TokenRecord:
        record = TokenRecord(
            timestamp=time.time(),
            source=source,
            error_id=error_id,
            error_type=error_type,
            prompt_tokens=usage.prompt_tokens,
            completion_tokens=usage.completion_tokens,
            total_tokens=usage.total_tokens,
            context_window_pct=round(usage.prompt_tokens / self._context_limit * 100, 1),
        )
        self._records.append(record)
        return record
    
    @property
    def total_tokens(self) -> int:
        return sum(r.total_tokens for r in self._records)
    
    @property
    def total_cost_estimate(self) -> float:
        """估算成本（仅供参考，qwen3:32b 约 ¥4/1M tokens）"""
        return self.total_tokens / 1_000_000 * 4.0
    
    def to_json(self) -> dict:
        return {
            "context_limit": self._context_limit,
            "total_records": len(self._records),
            "total_tokens": self.total_tokens,
            "estimated_cost_cny": round(self.total_cost_estimate, 2),
            "records": [dataclasses.asdict(r) for r in self._records],
        }
    
    def save(self, path: str = "output/token_usage.json"):
        with open(path, "w", encoding="utf-8") as f:
            json.dump(self.to_json(), f, ensure_ascii=False, indent=2)
```

每个 error 修复完成后，在 correction_report.json 中附带 token 消耗：

  "token_usage": {

  }
}
```

### 方向 B — 多 Agent 协作架构

**目标**：将单 agent 拆分为多个职责清晰的子 agent，各自独立优化。

#### 设计原则（来自 novelSpeakerV4 的经验）

1. **每个 agent 只有一项核心职责**
2. **agent 间通过结构化数据通信（不是自由对话）**
3. **每个 agent 的 prompt 极致精简**
4. **有独立的 memory 层管理上下文**

#### 新的 Agent 架构

```
┌─────────────────────────────────────────────────┐
│                 Boss (Orchestrator)               │
│  管理流程、调度 agent、判断终止条件                │
├─────────────────────────────────────────────────┤
│                                                    │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐     │
│  │ Detector   │  │ Corrector  │  │ Verifier   │     │
│  │ Agent      │  │ Agent      │  │ Agent      │     │
│  │            │  │            │  │            │     │
│  │ 只检测     │  │ 只生成     │  │ 只验证     │     │
│  │ 错误       │  │ 候选并判断  │  │ 修正质量   │     │
│  └───────────┘  └───────────┘  └───────────┘     │
│                                                    │
│  ┌───────────┐  ┌───────────┐                     │
│  │ Quality    │  │ Reporter   │                     │
│  │ Agent      │  │ Agent      │                     │
│  │            │  │            │                     │
│  │ 终局质量   │  │ 生成报告   │                     │
│  │ 验收       │  │ 含 token   │                     │
│  └───────────┘  └───────────┘                     │
│                                                    │
│  ┌──────────────────────────────────────┐          │

│  │ 全局记录所有 LLM 调用的 token 消耗     │          │

#### Agent 详细职责

##### 1. Detector Agent（改造现有的 DetectorPipeline）

当前已有，但可以增加一步：**错误分类与优先级排序**。

```python
class DetectorAgent:
    """
    职责：运行规则检测器 → 输出带优先级的 ErrorQueue
    
    改动点：
    1. 对每个错误计算"置信度"（consecutive=高, wrong_symbol=中, long_dialogue=低）
    2. 按置信度排序（高级优先处理）
    3. 附加"关联错误"信息（相邻位置的不同检测器结果可能相关）
    4. 记录检测耗时、覆盖范围
    """
```

##### 2. Corrector Agent（拆分现有的 CandidateDecisionAgent）

```python
class CorrectorAgent:
    """
    职责：接收 ErrorRecord → 生成候选 → 送 LLM 判断 → 应用修正
    
    与原 CandidateDecisionAgent 的区别：
    1. 只负责"修正"这一个步骤，不关心检测和验证
    2. 修正后返回 (ErrorRecord, AppliedCandidate, TokenUsage)
    3. 不直接写 ErrorQueue，返回结果由 Boss 处理
    4. LLM 调用的 token 消耗通过 TokenTracker 记录
    
    子模块：
    - CandidateGenerator（已有，增强）
    - LLM Judge（判断选哪个候选，增强 prompt 带 token 预算提醒）
    """
```

##### 3. Verifier Agent（已有，但需要加强启用）

当前 Verifier 在 candidate-decision 路径中几乎不用（候选由规则生成，认为不需要验证）。但这是错误的——**规则生成也可能引入新错误**。

```python
class VerifierAgent:
    """
    职责：验证一个修正是否真的修复了问题，且没有引入新问题
    
    验证维度：
    1. 符号平衡：修正后「」数量是否平衡
    2. 无连续符号：修正后没有新引入的「「或」」
    3. 长度变化合理：替换前后长度变化在预期范围内
    4. 语义保留（可选 LLM 验证）：修正没有改变原文意思
    
    输出：VerifierVerdict (pass/fail/uncertain)
    
    对 rule-precheck 路径的改动：
    当前 rule-precheck 已经跑了一遍完整检测（验证错误数降低），
    但缺少单次修正的验证。建议：
    - rule-precheck 通过的候选仍需经过快速规则验证（1-3）
    - rule-precheck 不通过的候选走 LLM 判断 → 验证
    """
```

##### 4. Quality Agent（新增）

```python
class QualityAgent:
    """
    职责：每轮结束后做全文质量评估，不修正，只评估
    
    评估维度（参考开发方案中的"硬性指标"）：
    1. 「」数量平衡
    2. 无连续相同符号
    3. 无非标准对话符号
    4. 无超长对话
    
    额外维度：
    5. 错误数收敛趋势（本轮 vs 上轮）
    6. token 消耗趋势（是否过度消耗）
    
    输出：QualityReport → Boss 判断是否终止或继续
    """
```

##### 5. Boss (Orchestrator) — 升级现有的 run_pipeline

```python
class Boss:
    """
    职责：编排整个纠错流程，调度子 agent，判断终止
    
    流程：
    Phase 0: TextLoader 加载文本
    Phase 1: DetectorAgent.run() → ErrorQueue
    Phase 2: multi-round correction loop
      - CorrectorAgent.run(error) → fixed/skipped/failed
      - VerifierAgent.verify(correction) → pass/fail/uncertain
      - TokenTracker.record(...)  # 每步都记录
    Phase 3: QualityAgent.evaluate(text) → QualityReport
    Phase 4: 收敛判断（同现有逻辑）
    Phase 5: ReporterAgent.generate() → 含 token 消耗的全量报告
    
    新增功能：
    - 每轮结束后打印 token 统计摘要
    - 上下文使用率预警（>80% 时提醒）
    - 支持 --token-report 参数单独输出 token 日志
    """
```

### 方向 C — 上下文窗口充分利用

**目标**：让每次 LLM 调用都能充分利用 qwen3:32b 的 40K 上下文窗口。

#### 当前问题

LLM fallback 的 calls 非常简短：

```
User prompt: 200-300 行 JSON（描述错误 + 候选列表）
System prompt: 5-8 行规则
LLM response: 1 行 JSON 或 tool call
```

上下文使用率估算：`(300 + 5 + 50) / 40960 ≈ 0.9%`

这意味着：
- 模型只看到极少量上下文
- 无法进行深度语义理解
- 只能靠候选生成器的质量，LLM 沦为"yes/no 分类器"

#### 改进方案

**1. 为 LLM 调用提供更丰富的上下文**

```python
def _user_prompt(self, error, candidates):
    return {
        "error": {
            "error_id": error.error_id,
            "error_type": error.error_type,
            "line_number": error.line_number,
        },
        "context": {
            "before": error.context_before[-800:],   # 800 chars before
            "after": error.context_after[:800],      # 800 chars after
            "full_sentence": extract_sentence(error),  # 完整句子上下文
        },
        "candidates": [...],
        "surrounding_dialogues": extract_nearby_dialogues(error, n=3),  # 附近3段对话
    }
```

目标：每次 LLM 调用使用 **3,000-8,000 tokens** 的上下文，占 40K 窗口的 **7-20%**。
既不是 1%（浪费），也不是 100%（溢出风险）。

**2. 对复杂错误（long_dialogue / missing_bracket）提供更多上下文**

这些错误类型需要理解全文角色关系，可以给 10K+ 上下文。

```python
CONTEXT_BUDGET = {
    "consecutive": (200, 200),        # 短上下文，符号修复足够
    "wrong_symbol": (500, 500),       # 中等上下文，判断是否为对话符号
    "unpaired": (500, 500),
    "long_dialogue": (2000, 2000),    # 长上下文，需要看完整段
    "missing_bracket": (2000, 2000),
}
```

**3. 上下文使用率预警**

```python
class ContextMonitor:
    def check(self, usage_pct: float, error_type: str):
        if usage_pct < 2.0:
            # 警告：上下文利用率过低，可能应该给更多上下文
            pass
        elif usage_pct > 80.0:
            # 严重警告：上下文即将溢出，可能需要分段
            pass
```

### 方向 D — 验证与质量保障

#### D1. 启用 Verifier 对规则候选的验证

当前 rule-precheck 确认"整体错误数降低"就直接应用候选，缺少单次修正质量检查。

建议：

```python
def _apply_with_verify(self, error, candidate):
    """应用候选 + 验证"""
    # 1. 保存修正前的快照
    snapshot = self._text.text
    
    # 2. 应用修正
    self._text.replace_range(candidate.start_offset, candidate.end_offset, candidate.replacement)
    
    # 3. 快速规则验证（不调 LLM）
    quick_checks = self._quick_verify(error, candidate)
    if not quick_checks.all_pass():
        self._text.text = snapshot  # 回滚
        return AgentResult(verdict="fail", reason=quick_checks.failure_reason())
    
    # 4. (可选) LLM 语义验证
    if self._llm_fallback:
        verdict = self._verifier.verify(error, candidate)
        if verdict != "pass":
            self._text.text = snapshot  # 回滚
            return AgentResult(verdict=verdict, ...)
    
    # 5. 确认修正成功
    return AgentResult(verdict="pass", ...)
```

### 方向 E — 检测器覆盖度扩展（先行）

**目标**：消除 158 个残留非标准符号的检测盲区，使修正后的「」数量逼近答案的 1349 对。

#### 根因分析

结合上文的对比数据，158 个残留符号的漏检原因分类：

| 漏检模式 | 估计数量 | 根因 |
|---------|---------|------|
| `「」` 内部嵌套的 `[` `]` | ~70 | WrongSymbolDetector 的嵌套保护过滤：`count_nested()` 检测到在 `「」` 内部就跳过 |
| 短注释型 (`[1]` `[注]`) | ~40 | SmartSkip 逻辑（`_is_likely_comment()`）误判为注释跳过 |
| 不成对残留（修复后产生的新不平衡）| ~25 | 修复一个括号导致另一侧失衡，但检测器认为"已处理"就不再抓 |
| 弯引号 `""` | ~4 | 弯引号检测覆盖不全 |
| 其他（分页标记、数字编号等）| ~19 | 特殊格式，当前检测器无匹配模式 |

#### 解决策略

**1. 修复嵌套保护逻辑**

当前代码（`src/detector/wrong_symbol.py` 中的 `count_nested()`）：
```python
def count_nested(self, text, pos):
    # 统计从 pos 向前/向后还有多少层「」，
    # 如果 > 0 说明在「」内部，跳过检测
```

问题：`「...[...]...」` 中，`[...]` 内部的 `[]` 被正确跳过（因为它在 `「」` 外部），但如果是 `「他说这是[重要的事]」`，`[` 在 `「」` 内部——虽然它仍然是需要修正的符号。

建议改成：**取消 `「」` 内部的跳过**。所有非标准符号都应被标记，无论是否在 `「」` 内部。但如果在一个大段 `「」` 内部出现成对的 `[]`，可以降低优先级（P2 而非 P1），让 LLM 判断是否真的是注释。

**2. 缩小 SmartSkip 范围**

当前 SmartSkip 跳过所有 `[数字]` 和 `[短文本]`，但小说对话中 `[重要的事]` 这样的短文本也需要修正。

- 如果差异 > 可接受阈值（如 > 5），追加扫描上次修正造成的不平衡
- 生成新的 ErrorRecord 让下一轮处理
